好，下面我把前面讨论过的内容和“实用功能点”一起融合，整理成一版 **PretextChat Roadmap 正式功能表**。
我会按 **Phase 1 / Phase 2 / Phase 3** 来写，并给每个功能标注：

- **优先级**：P0 / P1 / P2
- **理由**：为什么放在这一阶段

我先约定一下优先级含义：

- **P0**：没有它，这个阶段目标就不成立
- **P1**：强烈建议做，能明显增强体验，但不是阶段成立的绝对前提
- **P2**：可以做，但更偏增强或差异化，不应阻塞主线

# PretextChat Roadmap 正式功能表

## Phase 1：验证产品成立

### 阶段目标

验证用户是否愿意用 PretextChat 替代普通浏览器中的 AI 标签页使用方式，把它作为 **AI 主入口 + 多实例管理器**。

### 核心判断标准

- 用户是否愿意从这里进入 AI
- 用户是否会持续打开多个 AI 实例
- 用户是否会依赖“恢复上次状态”
- 用户是否认为这比普通浏览器更适合管理 AI 任务

### 功能表

| 功能                                         | 优先级 | 理由                                                                                 |
| -------------------------------------------- | ------ | ------------------------------------------------------------------------------------ |
| 预置 AI 应用目录                             | P0     | 这是整个产品的起点。MVP 必须开箱即用，不能让用户输入网址、理解白名单、自己配置站点。 |
| AI 应用卡片点击即创建实例                    | P0     | 这是“AI 应用”到“任务实例”的第一步转化，没有这个动作，产品就只是静态目录。            |
| 多实例管理：新建实例                         | P0     | PretextChat 的核心价值之一就是“同一 AI 可多开”，没有新建实例就没有差异化。           |
| 多实例管理：切换实例                         | P0     | 用户必须能在多个实例之间自然切换，否则无法形成工作流。                               |
| 多实例管理：关闭实例                         | P0     | 实例是主对象，关闭是基本生命周期动作。                                               |
| 多实例管理：重命名实例                       | P0     | 用户必须把实例理解为“任务窗口”，而不是“第二个网页”，重命名是关键。                   |
| 基础标签页系统                               | P0     | 实例需要一个清晰的可见导航层，标签页是最自然的承载方式。                             |
| 当前标签高亮与快速切换                       | P0     | 没有清晰激活态，用户会立刻迷路。                                                     |
| 最近一次会话恢复                             | P0     | 这是产品是否能从“工具”变成“主工作台”的关键功能。                                     |
| 登录态保持                                   | P0     | 如果用户每次重新打开都要重新登录各 AI 网站，产品价值会明显下降。                     |
| 最近使用的 AI                                | P1     | 让再次打开常用 AI 更快，明显提升高频使用体验。                                       |
| 快速重开最近实例                             | P1     | 比“恢复整个会话”更轻，也更适合用户临时回到某个任务。                                 |
| 启动方式选择：进入首页 / 恢复上次会话        | P1     | 这是很实用的产品细节，能照顾不同使用习惯。                                           |
| 实例默认命名策略（如 ChatGPT 1 / ChatGPT 2） | P1     | 用户第一眼就能看懂多个实例，不会乱。                                                 |
| 应用图标 + 名称统一视觉规范                  | P1     | 这是基础易用性，不复杂，但很有助于识别。                                             |
| 最近关闭实例恢复                             | P2     | 很实用，但不必阻塞首版上线。可以作为首版后半段补强。                                 |
| 应用收藏 / 常用应用置顶                      | P2     | 有价值，但 Phase 1 先验证主链路即可。                                                |

## Phase 1 的主用户流程

### 流程 1：打开 AI

1. 用户启动 PretextChat
2. 看到预置 AI 应用目录
3. 点击 ChatGPT
4. 系统创建一个 ChatGPT 实例
5. 顶部出现对应标签
6. 用户开始使用

### 流程 2：同一 AI 多开

1. 用户已经打开一个 ChatGPT 实例
2. 再次点击 ChatGPT
3. 系统创建第二个 ChatGPT 实例
4. 顶部新增标签
5. 用户将它重命名为“代码助手”
6. 两个实例独立使用

### 流程 3：恢复工作状态

1. 用户关闭客户端
2. 下次重新打开
3. 首页显示“恢复上次会话”或自动恢复
4. 系统恢复最近一次实例集合与激活状态
5. 用户继续工作

# Phase 2：进入工作台阶段

### 阶段目标

把产品从“AI 标签管理器”推进到“AI 并行工作台”，开始体现 Pretext 的交互性能价值和重度用户效率价值。

### 核心判断标准

- 用户是否开始在 PretextChat 内持续停留工作，而不是只把它当入口
- 用户是否开始并行使用多个 AI
- 用户是否明显感知到切换、输入、长内容场景更顺

### 功能表

| 功能                                     | 优先级 | 理由                                                                  |
| ---------------------------------------- | ------ | --------------------------------------------------------------------- |
| 工作区与分屏：单栏 / 双栏                | P0     | 这是从“管理多个实例”走向“并行使用多个实例”的关键一步。                |
| 双栏模式下实例分配                       | P0     | 有分屏还不够，用户必须能把不同实例放入左右栏。                        |
| 统一 Prompt Bar：发送到当前实例          | P0     | 这是产品层的统一输入入口，能增强“工作台感”，但先只发给当前实例。      |
| 长会话场景稳定性优化                     | P0     | 这是 Pretext 应体现价值的阶段：长输入、切换、resize、长内容场景更顺。 |
| 标签拖拽排序                             | P1     | 当实例变多后，整理顺序会变得很重要。                                  |
| 固定标签                                 | P1     | 高频任务实例应能固定，提高重度使用效率。                              |
| 最近关闭实例恢复                         | P1     | 到这个阶段已经很高频了，这个功能会变得很实用。                        |
| 快捷键体系                               | P1     | 能显著增强“工具感”和高频使用效率。                                    |
| Command Palette                          | P1     | 重度用户会很喜欢，用来快速打开应用、切换实例、执行命令。              |
| 标签处理中提示 / 未读提示                | P1     | 多实例并行时，用户需要感知哪个实例正在生成结果。                      |
| 应用收藏 / 置顶                          | P1     | 当支持的 AI 越来越多时，这会影响效率。                                |
| Prompt 草稿短暂保留                      | P2     | 有价值，但不应过早把输入系统做复杂。                                  |
| 分屏比例拖拽                             | P2     | 有帮助，但单纯双栏先成立更重要。                                      |
| 更丰富的启动页信息（最近任务、常用实例） | P2     | 属于体验增强，不是主链路核心。                                        |

## Phase 2 的主用户流程

### 流程 4：并排使用两个 AI

1. 用户打开 ChatGPT 实例 A
2. 打开 Claude 实例 B
3. 点击进入双栏模式
4. A 显示左侧，B 显示右侧
5. 用户同时查看两个 AI 的结果

### 流程 5：统一输入到当前实例

1. 用户当前激活某个实例
2. 在底部 Prompt Bar 输入内容
3. 点击发送
4. 系统注入当前实例
5. 用户继续在当前工作台使用

### 流程 6：高频切换

1. 用户打开多个实例
2. 使用快捷键或 Command Palette 切换
3. 标签提示当前状态
4. 用户更少依赖鼠标点击

# Phase 3：形成差异化壁垒

### 阶段目标

让 PretextChat 从“好用的 AI 工作台”变成“有明显壁垒的 AI 工作操作系统”。

### 核心判断标准

- 用户是否开始依赖 PretextChat 进行复杂 AI 工作编排
- 用户是否愿意长期沉淀自己的 AI 使用结构
- 产品是否与普通 AI 客户端明显拉开差距

### 功能表

| 功能                                | 优先级 | 理由                                               |
| ----------------------------------- | ------ | -------------------------------------------------- |
| 一问多发                            | P0     | 这是非常强的差异化能力，真正体现“并行 AI 工作流”。 |
| 多实例结果对比视图                  | P0     | 一问多发之后必须有更好的比较承载方式。             |
| 多工作区                            | P0     | 用户会开始按任务类型组织自己的 AI 结构。           |
| 实例分组                            | P1     | 当实例越来越多后，需要更清晰的归类方式。           |
| 独立窗口 / 多窗口                   | P1     | 高阶用户会希望把实例真正拆成多个工作窗口。         |
| 自定义 AI 应用入口                  | P1     | 到这个阶段才适合开放，避免 MVP 阶段失焦。          |
| 团队级应用预设                      | P1     | 有助于团队协作与标准化。                           |
| Prompt 模板                         | P1     | 当产品进入重度工作流阶段，模板才真正有意义。       |
| 实例模板                            | P1     | 用户可快速创建“研究助手”“写作助手”等固定结构。     |
| 工作区模板                          | P1     | 提高复杂工作流程复用效率。                         |
| 本地导入 / 分享应用配置             | P2     | 偏生态能力，不是最先要做。                         |
| 跨设备同步                          | P2     | 很有价值，但工程复杂度高，应放后。                 |
| 高级文本性能缓存与更深 Pretext 策略 | P2     | 属于壁垒深化，但建立在前面主功能被验证之后。       |

---

## Phase 3 的主用户流程

### 流程 7：一问多发并对比

1. 用户输入一段 prompt
2. 选择多个目标实例
3. 一键发送
4. 各实例分别处理
5. 系统进入对比视图
6. 用户横向比较结果

### 流程 8：按工作区组织

1. 用户创建“写作工作区”
2. 放入 ChatGPT、Claude、Gemini 各自实例
3. 再创建“代码工作区”
4. 在不同工作区之间切换
5. 每个工作区保留自己的状态

# 最终优先级总结

## Phase 1 必须做完的 P0

这是最小成立集合：

1. 预置 AI 应用目录
2. 点击 AI 应用即创建实例
3. 新建 / 切换 / 关闭 / 重命名实例
4. 基础标签页系统
5. 最近一次会话恢复
6. 登录态保持

这 6 个点决定 PretextChat 能不能成立。

---

## Phase 2 最该做的 P0

这是从“能用”走向“像工作台”的关键集合：

1. 单栏 / 双栏工作区
2. 双栏实例分配
3. 统一 Prompt Bar（仅发给当前实例）
4. 长会话场景稳定性优化

---

## Phase 3 最能形成壁垒的 P0

这是从“好用”走向“难替代”的关键集合：

1. 一问多发
2. 多实例对比视图
3. 多工作区

---

# 我的建议结论

如果你现在准备真正推进产品，我建议你把 roadmap 当成三句话来记：

## Phase 1

**先证明用户愿意用它打开和管理 AI。**

## Phase 2

**再证明用户愿意在里面长时间并行工作。**

## Phase 3

**最后再做出别人不容易替代的 AI 工作流能力。**

下一步最合适的是把这份 roadmap 再压成一个更利于团队对齐的版本，例如：

**功能名｜阶段｜优先级｜一句话理由｜是否阻塞上线**

这样会更像产品排期表。
